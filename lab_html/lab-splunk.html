AI-Powered SIEM with Splunk

Install Splunk Free, ingest real security logs, write SPL detection queries, build dashboards, and use AI to write correlation rules that automate threat detection.

⏱ 90 min
🖥️ Kali Linux or Host Machine
🔧 Splunk Free · SPL · Python
🤖 AI Query Builder
📊 Lab Scenario

Your organization's SIEM is producing thousands of raw alerts per day with no prioritization. You've been asked to build ML-based correlation rules that reduce noise and surface real threats. You'll use Splunk Free with sample security logs to build detection logic, then use AI to write and refine SPL queries.

Phase 0 — Install & Configure Splunk
Get Splunk Free running and ingest sample security logs
Download and install Splunk Free
Free tier supports 500MB/day — more than enough for labs
Download from splunk.com/en_us/download/splunk-enterprise.html (requires free account). Install on your Kali VM or directly on your Windows host.
# Linux install
wget -O splunk.deb 'https://download.splunk.com/products/splunk/releases/9.2.0/linux/splunk-9.2.0-deb.deb'
sudo dpkg -i splunk.deb
sudo /opt/splunk/bin/splunk start --accept-license
# Then navigate to http://localhost:8000
Download the Boss of the SOC (BOTS) sample dataset
BOTS is a free, realistic security dataset used by SOC analysts worldwide
The BOTS dataset contains real attack logs: web attacks, malware, AD attacks, and more. Download from GitHub:
git clone https://github.com/splunk/botsv3.git
# Upload the data via Splunk UI:
# Settings → Add Data → Upload → select botsv3 files
# Or use the Splunk Add-on for BOTS from Splunkbase (free)
💡
Alternatively, generate your own logs by running attacks against Metasploitable from Kali while logging with rsyslog or Windows Event Forwarding.
Verify data is searchable
Confirm Splunk has indexed your logs before building detections
# In Splunk Search (http://localhost:8000/en-US/app/search/search)
index=* | head 20 # Show first 20 events
index=* | stats count by sourcetype # List all sourcetypes
index=* | stats count by index # List all indexes with counts

Phase 1 — Threat Hunting with SPL
Write SPL queries to detect real attack patterns
Detect brute force attacks
Find authentication failures indicating credential stuffing
# Detect brute force: 10+ failed logins from same IP in 5 min
index=* (EventCode=4625 OR "authentication failure" OR "Invalid user")
| bucket _time span=5m
| stats count by src_ip, user, _time
| where count > 10
| sort -count
🤖 AI Analyst

Paste your query results. The AI will identify the source IPs, explain the attack pattern, and suggest SPL refinements to reduce false positives.

Detect lateral movement
Find internal hosts making unusual connections to other hosts
# Lateral movement: internal IPs connecting to multiple hosts
index=* sourcetype=syslog OR sourcetype=firewall
| where src_ip!="0.0.0.0" AND match(src_ip,"^192\.168\.")
| stats dc(dest_ip) as unique_dests, values(dest_port) as ports by src_ip
| where unique_dests > 5
| sort -unique_dests
Detect DNS tunneling
High-volume or long DNS queries can indicate C2 communication
# DNS tunneling indicators: long subdomain queries or high frequency
index=* sourcetype=dns OR sourcetype="stream:dns"
| eval query_length=len(query)
| where query_length > 50
| stats count, avg(query_length) as avg_len by src, dest, query
| sort -count

Phase 2 — Dashboards & Alerts
Turn queries into automated alerts and security dashboards
Create a SOC security dashboard
Visualize key security metrics in one view
In Splunk: Dashboards → Create New Dashboard → Classic Dashboards. Add panels using your SPL queries. Recommended panels:
# Panel 1: Failed logins over time (time chart)
index=* EventCode=4625 | timechart count span=1h

# Panel 2: Top attacking IPs (bar chart)
index=* EventCode=4625 | stats count by src_ip | sort -count | head 10

# Panel 3: Geographic source map (choropleth)
index=* | iplocation src_ip | geostats count by Country
Set up a real-time alert
Trigger notifications when brute force threshold is exceeded
From your brute force search: Save As → Alert. Configure: Schedule = Real-time, Trigger when = Number of results > 0, Action = Send email or webhook.
🤖 AI Analyst

Tell the AI what alert you're building and how many false positives you're seeing. It will suggest threshold tuning, exclusion lists, and risk-based scoring to reduce noise.

Phase 3 — AI-Written Detection Rules
Use AI to generate advanced SPL correlations you couldn't easily write manually
AI generates composite attack detection
Multi-stage detections that correlate events across different log sources
🤖 AI Analyst — SPL Generation

Describe the attack scenario (e.g., "brute force → successful login → new user created → data exfiltration within 30 minutes"). The AI will write a correlated SPL query that detects the full attack chain across multiple log sources.

Implement Splunk MLTK anomaly detection
Use machine learning to baseline normal behavior and flag deviations
# Install Splunk MLTK (free from Splunkbase)
# Then use fit/apply commands for anomaly detection:
index=* sourcetype=access_combined
| timechart span=1h count as requests
| fit DensityFunction requests by date_hour dist=auto into traffic_model
| eval isAnomaly=if(BoundaryRanges=="[-Inf, Inf]",0,1)
| where isAnomaly=1
Export detections as Sigma rules
Convert your Splunk detections to vendor-neutral Sigma format
🤖 AI Analyst

Paste your most effective SPL detection query. The AI will convert it to a Sigma rule in YAML format that works with Elastic, QRadar, Microsoft Sentinel, and other SIEMs.
